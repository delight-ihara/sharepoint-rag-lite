## 1. 概要

### 1.1 背景と目的

SharePoint 上に経営資料・人事労務規程・営業関連文書を格納しているが、以下の課題がある。

- 社員が必要な情報にたどり着くまでに時間がかかる
- 経営層が意思決定時に関連文書を横断的に参照する手段がない
- 文書の閲覧権限が未設定で、機密文書が全社員に公開されている

これらを解決するため、SharePoint 文書を対象とした ACL 付き RAG チャットボットを構築する。

既存の Azure ネイティブ構成（Azure AI Search 使用）は動作確認済みだが、AI Search のコスト（月額 ¥13,000+）がスタートアップ規模には割高であるため、AI Search を使わないLite 構成で同等の機能を実現する。

| 区分 | 内容 |
|------|------|
| 主目的 | AI Search を使わず、既存構成と同等の ACL 付き RAG をスタートアップ規模向けで実現する |
| 事業目的 | まず社内で実用し、検証後に外販（SaaS）として展開する |
| 技術習得 | スタートアップ規模向け RAG の設計・構築パターンを確立し、提案テンプレート化する |

### 1.2 既存構成との関係

本構成は既存の [sharepoint-rag-azure](https://github.com/delight-ihara/sharepoint-rag-azure)（Azure AI Search 構成）の**代替実装**であり、同一の SharePoint データソース・同一の ACL 要件を満たす。

| 項目 | 既存構成 | 本構成 |
|------|----------|--------|
| ベクトル検索 | Azure AI Search | PostgreSQL + pgvector |
| ドキュメント取り込み | AI Search インデクサー + DI Layout | Graph API + Python（既存 sp_to_blob.py 流用） |
| ACL | AI Search フィルタ + Blob メタデータ | pgvector テーブルの allowed_groups カラム |
| LLM | Azure OpenAI | Azure OpenAI（同一） |
| 会話履歴 | Cosmos DB | PostgreSQL（同一 DB 内） |
| 認証 | Entra ID SSO | Entra ID SSO（同一） |
| 月額コスト | ¥13,000+ | 目標 ¥5,000 以下 |

既存コードのうち以下を流用する:
- `sp_to_blob.py`: SharePoint → テキスト抽出 + ACL 取得ロジック（Graph API permissions）
- `search.py`: ハイブリッド検索のクエリ構成ロジック（OData → SQL に書き換え）
- `llm.py`: 回答生成 + citation 抑制ロジック
- `server.js`: ユーザー認証ヘッダー抽出ロジック
- `11-test-spec.md`: ACL テスト仕様（18ケース）

### 1.3 既存構成に対する劣後点

AI Search を外すことで失われる機能・品質と、その緩和策を明記する。本構成の設計判断・テスト計画はこの表に基づく。

| # | 領域 | 既存構成（AI Search） | 本構成（pgvector） | 影響度 | 緩和策 |
|---|------|----------------------|-------------------|--------|--------|
| D-01 | **セマンティックリランカー** | MS 製リランカー内蔵。検索結果をクエリとの意味的関連度で再順位付け（`@search.rerankerScore`） | なし（ベクトル類似度のみ） | **高** — 関連度の低いチャンクが上位に残り、回答精度が下がる可能性 | Cohere Rerank API（無料枠 1,000クエリ/月）で代替可能。まずリランカーなしで精度を測定し、評価質問 70% 未達なら追加 |
| D-02 | **ハイブリッド検索の RRF 融合** | ベクトル検索（k=50, weight=2.0）+ キーワード検索を AI Search が RRF（Reciprocal Rank Fusion）で自動統合 | pgvector はベクトル検索のみネイティブ。キーワード検索（PostgreSQL 全文検索）との統合は自前実装が必要 | **中** — RRF なしだと固有名詞・型番など「意味」より「文字列一致」が重要なクエリで精度が落ちる | PostgreSQL の `ts_rank` + pgvector の `<=>` を SQL で結合する RRF を実装（約30行）。または LlamaIndex のハイブリッド検索機能を利用 |
| D-03 | **Document Intelligence Layout** | AI Search スキルセットで DI Layout（v4.0）を使用。表・見出し・段落の構造を保ったチャンキング | DI Layout は使用しない。テキスト抽出は Python ライブラリ（unstructured / PyMuPDF 等）で行う | **中** — 表形式データ（Excel 等）のチャンク品質が下がる可能性。既存でも GPT-4o-mini の Excel 精度は課題（R-03） | PDF/Word はテキスト抽出で十分。Excel は既存構成でも精度課題あり。DI Layout を API 単体で呼び出す構成も選択肢（$10/1,000ページ、100文書なら ~¥200 で済む） |
| D-04 | **エンベディングモデル** | text-embedding-3-large（3072次元） | text-embedding-3-small（1536次元）を想定 | **低〜中** — large と small で検索精度に差が出る可能性。ただし small でも MTEB ベンチマークで十分な精度 | コスト差は小さい（large: $0.13/1M tokens vs small: $0.02/1M tokens）。100文書なら月額数円の差。精度要件次第で large に切り替え可能 |
| D-05 | **マネージドインデクサー** | AI Search のインデクサーが Blob 検知 → パース → チャンキング → エンベディング → インデックス更新を自動実行 | 全パイプラインを自前で構築（Graph API → テキスト抽出 → チャンク → エンベディング → DB 書き込み） | **中** — 開発・保守工数が増える。インデクサーの障害検知・リトライも自前 | 既存の sp_to_blob.py + sync.py のロジックを流用。パイプラインは Python スクリプト 1本にまとめられる。スケジュール実行は Azure Functions Timer Trigger or cron |
| D-06 | **ACL の保守責任** | AI Search のセキュリティトリミングは MS が保守。SP の権限モデル変更に追従 | 自前の ACL フィルタ（SQL WHERE 句）を保守。SP 側の権限モデル変更に自分で追従する必要がある | **中** — SP の権限 API 仕様変更時にフィルタが壊れるリスク | 既存の sp_to_blob.py は Graph API v1.0（安定版）を使用。PoC では個人権限のみ（セキュリティグループ未使用）で複雑さを抑える。ACL テスト 18 ケースで回帰検知 |
| D-07 | **Cosmos DB の会話履歴** | Cosmos DB（マネージド NoSQL）で会話履歴を管理。グローバル分散・自動スケール対応 | PostgreSQL テーブルで会話履歴を管理 | **低** — 10ユーザー規模では PostgreSQL で十分。Cosmos DB の利点（グローバル分散）は PoC では不要 | PostgreSQL の JSONB 型で会話履歴を格納。スケール時も 300 ユーザーなら問題なし |

**総合判断**: 最も影響が大きいのは **D-01（リランカー）** と **D-02（RRF）**。この 2 点が検索精度に直結する。PoC ではまずリランカーなし・ベクトル検索のみで評価質問 10 問を実施し、70% 未達なら D-01（Cohere Rerank）→ D-02（RRF 実装）の順で追加する段階的アプローチを取る。

### 1.4 スコープ

| 項目 | 値 |
|------|-----|
| 対象システム | SharePoint RAG チャットボット（Lite 構成） |
| 対象ユーザー | 10名（PoC）→ 100-300名（スケール時） |
| 対象データ | SharePoint 文書 100件以上（PDF, Office）※既存と同一 |

### 1.5 ゴールとノンゴール

**ゴール（この構成で達成すること）**
- AI Search なしで、既存構成と同等の検索精度・ACL 制御を実現する
- 月額ランニングコスト ¥5,000 以下
- 外販を見据えた構成（クエリログ・テナント分離の拡張性）
- 既存の ACL テスト仕様（18ケース）を全パスする
- PoC → プロダクト品質への引き上げ（エラーハンドリング・ストリーミング・GraphRAG・CI/CD・監視）

**ノンゴール（意図的にスコープ外とするもの）**
- マルチテナント実装（外販フェーズで対応。本構成では拡張可能な設計に留める）
- モバイルアプリ / 社外向け UI
- UI/UX の最適化（既存と同等の最小チャット画面で十分）
- IaC テンプレート・構築自動化（Phase 1 で対応）

## 2. ユースケース

| ID | 名称 | アクター | 概要 |
|----|------|----------|------|
| UC-1 | 経営判断支援 | 経営層 | 経営に関する質問に対し、経営層限定文書を含む全文書から根拠リンク付きで回答する |
| UC-2 | 社員問い合わせ削減 | 一般社員 | 社内制度・規程に関する質問に対し、閲覧権限のある文書のみから回答する |
| UC-3 | 権限外アクセスの拒否 | 一般社員 | 権限外文書にのみ根拠がある質問に対し、情報を漏洩せず「該当なし」と応答する |
| UC-4 | クエリログ確認 | 管理者 | 誰がどんな問い合わせをしたかを確認できる |

### 2.1 回答の振る舞い

既存構成と同一。

- 根拠がある場合: 自然言語で回答 + 根拠リンク1〜3件
- 根拠が不足する場合: 「該当する情報が見つかりませんでした」と応答（推測しない）
- 会話は継続可能（文脈を踏まえた連続質問に対応）

## 3. 前提条件と制約

### 3.1 前提条件

| # | 前提 | 根拠 |
|---|------|------|
| P-01 | Azure を主軸クラウドとする | SP との親和性・既存環境との統一 |
| P-02 | 無料枠で収まるコンポーネントは Azure 以外も選択可 | コスト最適化 |
| P-03 | Azure AI Search は使用しない | コスト削減が本構成の目的 |
| P-04 | 既存の SharePoint データソース・フォルダ構成をそのまま使用する | 既存構成と同一環境 |
| P-05 | 既存コード（sp_to_blob.py 等）の ACL ロジックを流用する | 実装済み・テスト済み |
| P-06 | PoC は10名の最小構成。設計は100-300名スケールを想定 | 既存構成と同一 |
| P-07 | Azure サブスクリプション（従量課金）が利用可能 | 既存構成で確認済み |
| P-08 | Entra ID の管理者同意が取得可能 | 既存構成で取得済み |
| P-09 | **Entra ID アプリに ACL 展開用の権限が付与されていること** | `GroupMember.Read.All`, `Directory.Read.All`, `User.Read.All`, `Sites.Read.All`（サイトメンバー展開） が必須。詳細は [05-acl-remediation.md](05-acl-remediation.md)。`Sites.Read.All` が付与できない場合は `SP_SITE_MEMBERS` 環境変数でフォールバック可能 |
| P-10 | **SharePoint フォルダに匿名共有リンクがないこと** | anonymous scope のリンクがあると ACL が無効化される |

### 3.4 SharePoint 権限モデル

ACL はフォルダ単位で以下の4種類の権限エントリを処理する:

| 種類 | 例 | 処理 |
|------|------|------|
| 共有リンク (anonymous/organization) | 「リンクを知っているすべてのユーザー」 | `["*"]` として処理。`REJECT_ANONYMOUS_LINKS=true` で拒否可能 |
| SharePoint サイトグループ | Communication site Members | **制約あり**: Communication Site では Graph API で展開不可（`Sites.FullControl.All` が必要で権限過剰）。**推奨: Entra ID セキュリティグループで権限管理すること** |
| Entra ID グループ/ロール | Global Administrator | `/groups/{id}/transitiveMembers` or `/directoryRoles/{id}/members` で展開 |
| 個人ユーザー | user@example.com | メールアドレスをそのまま使用 |

## 8. リスクと残論点

| # | 論点 | 影響 | 対応 |
|---|------|------|------|
| R-01 | pgvector での検索精度が AI Search に劣る可能性（D-01〜D-04） | PoC 成否 | 段階的改善: ベクトルのみ → Cohere Rerank → RRF → embedding-large の順に追加 |
| R-02 | SP 権限モデルの複雑さ（継承・共有リンク・ネストグループ） | ACL 漏洩 | 既存 sp_to_blob.py のロジックを流用。PoC では個人権限のみ（既存と同一前提） |
| R-03 | 回答精度の再現性 | PoC 成否 | GPT-4o-mini の Excel 精度限界は既存と同一。チューニング継続 |
| R-04 | 無料枠サービスの制限・停止リスク | 可用性 | PoC 段階では許容。外販時に有料プランへ移行 |
| R-05 | 外販時のマルチテナント設計 | 事業拡大 | 本構成ではスコープ外。拡張可能な設計に留める |
| R-06 | GraphRAG のエンティティ抽出精度 | 回答品質 | GPT-4o-mini のエンティティ抽出は完璧ではない。誤抽出が回答に影響する可能性。段階的に評価 |
| R-07 | SP 削除ファイルのチャンク残留 | 検索品質 | 差分同期で削除が検知できないと、存在しないファイルのチャンクが回答に混入する |
| R-08 | トークン予算超過 | コスト・安定性 | 長い会話で履歴+チャンクがコンテキストウィンドウを超過するとエラーまたはコスト増 |
| R-09 | 間接プロンプトインジェクション | ACL バイパス・情報漏洩 | SP ドキュメント内に埋め込まれた悪意あるテキストがチャンクとして LLM に渡り、動作を操作される |
| R-10 | 依存パッケージの脆弱性 | セキュリティ | OSS ライブラリに CVE が発見された場合、PoC 期間中でも影響を受ける |

## 付録 B: コスト試算（概算）

アーキテクチャ設計書・リソース設計書で詳述する。

### PostgreSQL（pgvector）の構成選択肢

pgvector のホスティング先がコスト構造を決める最大の変動要因。

| 選択肢 | 月額 | 容量 | メリット | デメリット |
|--------|------|------|----------|-----------|
| **A: Supabase Free** | ¥0 | 500MB / pgvector 対応 | 無料。PoC には十分 | 500MB 上限。7日間アクセスなしで停止。バックアップなし |
| **B: Azure Database for PostgreSQL Flexible Server（B1ms）** | ~¥5,000 | 32GB | Azure 統一。バックアップ・監視あり | PoC 用途にはオーバースペック |

- PoC（10ユーザー・~5,000ベクトル）は **選択肢 A（Supabase Free）で十分**。500MB のうちベクトルデータは数十 MB 程度
- 外販・本番移行時は **選択肢 B に昇格**（データ移行は pg_dump/pg_restore で可能）

### 構成パターン別コスト

| パターン | pgvector | ホスティング | LLM | 合計 |
|----------|----------|-------------|-----|------|
| **PoC 最小** | Supabase Free (¥0) | Azure Container Apps Consumption (~¥0-500) | Azure OpenAI GPT-4o-mini (~¥100) | **~¥100-600/月** |
| **本番最小** | Azure PostgreSQL B1ms (~¥5,000) | Azure Container Apps Consumption (~¥0-500) | Azure OpenAI GPT-4o-mini (~¥100) | **~¥5,100-5,600/月** |

- PoC パターンは C-01 制約（¥5,000 以下）を大幅に下回る
- 本番パターンは C-01 制約の上限付近。PostgreSQL の SKU 選定はアーキテクチャ設計書で詳述
